一、技术栈与核心规范

1.1 技术栈选型

结合在线课程平台业务特性，适配前端技术栈，确定后端核心技术栈：

|层级|技术选型|说明|
|-|-|-|
|后端框架|Spring Boot 3 + Spring MVC|轻量级微服务框架，快速开发 RESTful API，适配前端请求范式|
|编程语言|Kotlin|强类型语言，生态完善，兼容主流中间件|
|数据库|MySQL 8.0 + Redis 7.0|MySQL 存储核心业务数据，Redis 缓存热点数据（课程列表、用户 token、学习进度）|
|认证授权|Spring Security + JWT|统一权限管控，支持 token 认证（适配前端登录态、管理员权限）|

1.2.2 编码规范

1.接口规范

* 所有接口返回统一格式：{ "code": 数字, "msg": "提示信息", "data": 数据 }，与前端响应拦截器适配。
* 接口路径命名：采用 RESTful 风格，如/api/v1/courses（课程列表）、/api/v1/courses/{id}（课程详情）。
* 请求方式：查询用GET，新增用POST，修改用PUT，删除用DELETE。



2.分层规范

* Controller 层：仅负责接收请求、参数校验、返回响应，不包含业务逻辑。
* Service 层：封装核心业务逻辑，按模块拆分（CourseService、UserService 等），通过接口 + 实现类解耦。
* Mapper 层：仅负责数据库操作，复杂 SQL 写在 XML 文件，简单 CRUD 用 MyBatis-Plus 内置方法。



3.数据传输规范

* 入参：前端请求参数封装为 DTO（Data Transfer Object），通过@Valid + 注解（@NotBlank、@NotNull）做参数校验。
* 出参：返回前端数据封装为 VO（View Object），屏蔽数据库实体敏感字段（如用户密码、创建时间）。



4.异常规范

* 自定义业务异常（BusinessException）处理业务错误（如 “课程不存在”、“订单已支付”）。
* 全局异常处理器（GlobalExceptionHandler）统一捕获异常，返回标准化错误响应。



5.命名规范

* 类名：PascalCase（如CourseController），方法名：camelCase（如getCourseDetail）。
* 常量：UPPER\_SNAKE\_CASE（如REDIS\_KEY\_USER\_TOKEN），数据库表名：lower\_snake\_case（如course\_info）。



二、核心模块设计

2.1 模块划分（对应前端页面 / 功能）

|模块名称|核心功能|关联前端页面|
|-|-|-|
|用户模块（User）|登录 / 注册、个人信息管理、权限校验、第三方登录（微信 / QQ）|登录页、注册页、个人中心页|
|课程模块（Course）|课程 CRUD、筛选 / 搜索、章节管理、课程评价|首页、课程列表页、课程详情页|
|订单模块（Order）|订单创建、支付回调、订单查询、退款处理|课程详情页（购买）、订单页|
|学习模块（Learning）|视频播放授权、学习进度记录、笔记 / 作业管理、讨论区|课程学习页|
|管理员模块（Admin）|课程审核 / 上下架、用户管理（封禁 / 解封）、数据统计、系统配置|管理员课程页、管理员用户页|
|通用模块（Common）|文件上传（头像 / 课程封面）、验证码生成、数据导出、缓存管理|所有涉及文件 / 验证码的页面|



2.2 核心模块实现思路

2.2.1 用户模块

* 认证流程：用户登录 → 验证账号密码 → 生成 JWT Token → 返回 Token + 用户基本信息 → 前端存储 Token，后续请求携带Authorization头。
* 权限管控：基于 Spring Security，区分 “普通用户”、“管理员” 角色，接口层面通过@PreAuthorize("hasRole('ADMIN')")限制访问。
* 数据安全：用户密码通过 BCrypt 加密存储，前端传输密码时建议做非对称加密（如 RSA）。

2.2.2 课程模块

* 课程筛选：接收前端筛选参数（分类、价格、评分），动态拼接 SQL 条件，结合 Redis 缓存热门课程列表（减轻数据库压力）。
* 视频资源：课程视频存储在 MinIO/OSS，返回前端时生成临时访问链接（带时效），防止资源盗链。
* 课程评价：支持分页查询评论、点赞 / 回复，评论内容做敏感词过滤。

2.2.3 订单模块

* 订单创建：生成唯一订单号 → 扣减课程库存（可选） → 创建订单记录 → 返回订单信息供前端支付。
* 支付回调：对接第三方支付（如支付宝 / 微信），异步接收支付结果 → 更新订单状态 → 同步学习权限（用户可访问课程）。
* 订单查询：支持按时间、状态筛选，关联查询课程名称、价格等信息，封装为 VO 返回。

四、部署与运维

4.1 多环境配置

* 开发环境（dev）：本地开发，连接测试数据库，开启接口文档、日志调试。
* 测试环境（test）：部署到测试服务器，连接测试数据库，供测试人员验证功能。
* 生产环境（prod）：部署到正式服务器，连接生产数据库，关闭调试功能，开启日志脱敏。



4.2 容器化部署

* 后端打包为 jar 包，通过 Dockerfile 构建镜像（基于 openjdk:17 镜像）。
* 与前端共用 docker-compose.yml，一键启动前端、后端、MySQL、Redis、MinIO 等服务。
* Nginx 做反向代理，区分前端静态资源和后端 API 请求（如/api转发到后端服务）。



4.3 性能优化

* 数据库：课程表、订单表加索引（如 course\_info 的 category\_id、order\_info 的 user\_id）；分页查询使用 MyBatis-Plus 分页插件，避免全表扫描。
* 接口：高频接口（课程列表）添加 Redis 缓存；大文件（视频）通过 CDN 加速。
* 并发：订单创建、支付回调等接口，添加分布式锁（Redis/Zookeeper），防止重复提交。
